A full end-to-end machine learning project that predicts whether a credit card customer will default on their next payment. Built with PyTorch, served via FastAPI, and visualised through a Streamlit dashboard.

## 📊 Dataset

**Source:** UCI Machine Learning Repository — Default of Credit Card Clients Dataset  
**Records:** 30,000 Taiwanese credit card customers  
**Target:** `default.payment.next.month` (1 = will default, 0 = will not default)  
**Class distribution:** ~78% no default / ~22% default (imbalanced)

### Original Features (23)

| Feature | Description |
|---|---|
| `LIMIT_BAL` | Credit limit balance (NT$) |
| `SEX` | 1 = Male, 2 = Female |
| `EDUCATION` | 1 = Graduate School, 2 = University, 3 = High School, 4 = Other |
| `MARRIAGE` | 1 = Married, 2 = Single, 3 = Other |
| `AGE` | Age in years |
| `PAY_0` to `PAY_6` | Repayment status for April–September (−2 = no consumption/no balance, −1 = paid on time, 0 = revolving credit, 1–9 = months delayed) |
| `BILL_AMT1` to `BILL_AMT6` | Bill statement amounts for April–September (NT$) |
| `PAY_AMT1` to `PAY_AMT6` | Previous payment amounts for April–September (NT$) |

## ⚙️ ML Pipeline (`Week3_Credit_Default.ipynb`)

### Step 1 — Data Cleaning
- Dropped the `ID` column (no predictive value)
- Merged undocumented `EDUCATION` codes (0, 5, 6) → 4 (Other)
- Merged undocumented `MARRIAGE` code (0) → 3 (Other)

### Step 2 — Feature Engineering (+20 features → 43 total)

| Group | Features Created | Purpose |
|---|---|---|
| Delay | `TOTAL_DELAY`, `MAX_DELAY`, `CURRENTLY_DELAYED`, `CONSECUTIVE_DELAY` | How severe and recent payment delays are |
| Bill | `AVG_BILL`, `UTILIZATION`, `BILL_TREND`, `MAX_BILL` | Credit usage and whether debt is growing |
| Payment | `TOTAL_PAID`, `AVG_PAID`, `PAY_TREND`, `ZERO_PAY_COUNT` | Payment behaviour and effort |
| Balance | `BAL_AFTER_PAY1`, `BAL_AFTER_PAY2`, `BAL_AFTER_PAY3`, `AVG_REMAINING_BAL`, `BAL_GROWING` | Net debt after payments |
| Demographic | `AGE_GROUP`, `HIGH_EDUCATION`, `LIMIT_PER_AGE` | Credit worthiness relative to profile |

### Step 3 — Train / Val / Test Split

| Set | Size | Purpose |
|---|---|---|
| Train | 70% (~21,000) | Model learns weights |
| Validation | 15% (~4,500) | Hyperparameter tuning, early stopping |
| Test | 15% (~4,500) | Final honest evaluation |

`stratify=y` ensures all three sets have the same ~22% default rate.

### Step 4 — Scaling
`StandardScaler` fitted **only on train set**, then applied to all three sets. Saved to `artifacts/scaler.pkl` for use in the API.

### Step 5 — Default Batch Size
Initial `BATCH_SIZE = 128` is set in Cell 17 for the default DataLoaders. The random search overrides this per trial (32, 64, or 128).

### Step 6 — Model Architecture (MLP)

```
Input (43)
  → Linear(43 → 128) → BatchNorm1d → ReLU → Dropout(0.3)
  → Linear(128 → 64) → BatchNorm1d → ReLU → Dropout(0.3)
  → Linear(64 → 32)  → BatchNorm1d → ReLU → Dropout(0.3)
  → Linear(32 → 1)   [raw logit — no activation]
```

| Component | Choice | Reason |
|---|---|---|
| Loss | `BCEWithLogitsLoss` with `pos_weight ≈ 3.5` | Handles class imbalance — penalises missed defaulters 3.5× more |
| Optimizer | `Adam` (default training) / `AdamW` (random search & retrain) | Adaptive learning rate; AdamW has cleaner weight decay |
| Activation | `ReLU` | Avoids vanishing gradients, fast to compute |
| Regularisation | `BatchNorm1d` + `Dropout(0.3)` | Stabilises training, prevents overfitting |
| Weight init | He (Kaiming) | Correct variance scaling for ReLU networks |
| Early stopping | `patience=7`, `min_delta=0.001` | Stops only when improvement exceeds 0.001; restores best weights |

### Step 7 — Experiment Phase 1: Fixed Experiments (6 runs)

Six fixed configurations are tested first using MLflow experiment `credit_default_experiments`:

| Run name | Init | Optimizer | LR | Weight Decay |
|---|---|---|---|---|
| he_adam | He | Adam | 1e-3 | 1e-4 |
| xavier_adam | Xavier | Adam | 1e-3 | 1e-4 |
| he_adamw | He | AdamW | 1e-3 | 1e-4 |
| he_adam_lr_low | He | Adam | 1e-4 | 1e-4 |
| he_adam_wd_high | He | Adam | 1e-3 | 1e-3 |
| xavier_adamw | Xavier | AdamW | 1e-3 | 1e-4 |

Best model from this phase selected by **validation AUC**.

### Step 8 — Experiment Phase 2: Random Search (10 trials)

Random search over 10 trials using MLflow experiment `credit_default_random_search`:

| Hyperparameter | Search Space |
|---|---|
| Learning rate | 1e-3, 5e-4, 1e-4 |
| Weight decay | 0.0, 1e-4, 1e-3 |
| Batch size | 32, 64, 128 |
| Dropout rate | 0.2, 0.3, 0.4 |

Best model selected by **validation recall** (catching actual defaulters is more important than overall accuracy).

### Step 9 — Retrain Best Config (captures loss history)

The best config from random search is retrained from scratch (up to 60 epochs, `EarlyStopping(patience=7, min_delta=0.001)`) to capture the full epoch-by-epoch loss and accuracy history for plotting curves.

### Step 10 — Evaluation & Plotting

Three plots are generated from the retrained best model:
- **Loss curve** — train vs val loss per epoch
- **Accuracy curve** — train vs val accuracy per epoch
- **ROC-AUC curve** — with AUC score on test set

Additional standalone plots:
- **Confusion matrix** — at threshold 0.5
- **Confidence distribution** — correct vs wrong predictions histogram

### Step 11 — Threshold Tuning

Default threshold is 0.5. Thresholds from 0.5 down to 0.25 are tested:

| Threshold | Accuracy | Precision | Recall |
|---|---|---|---|
| 0.50 | Higher | Higher | Lower |
| **0.35** | Balanced | Balanced | Better |
| 0.25 | Lower | Lower | Highest |

**Chosen threshold: 0.35** — saved to `artifacts/model_config.json`.

### Step 13 — Permutation Importance

Wraps the PyTorch model in a scikit-learn compatible `TorchModelWrapper` class and uses `sklearn.inspection.permutation_importance` to measure how much recall drops when each feature is shuffled.

Top features by impact on recall:

**Original features:** `BILL_AMT3`, `PAY_AMT1`, `BILL_AMT1`, `AGE`, `LIMIT_BAL`  
**Engineered features:** `ZERO_PAY_COUNT`, `AVG_PAID`, `MAX_BILL`, `LIMIT_PER_AGE`, `TOTAL_PAID`

### Step 14 — Confusion Matrix Preset Extraction

Real test set examples are extracted for each quadrant using:
- **TP** — highest probability among true positives (most confident correct default)
- **TN** — lowest probability among true negatives (most confident correct safe)
- **FP** — highest probability among false positives (model most wrong on safe customer)
- **FN** — lowest probability among false negatives (model most wrong on defaulter)

## 📌 Design Decisions

**Why BCEWithLogitsLoss over BCELoss?**  
Numerically more stable. Sigmoid is applied internally rather than as a separate step, avoiding floating point errors in the log computation.

**Why `min_delta=0.001` in EarlyStopping?**  
Without a minimum improvement threshold, early stopping can trigger on noise — a tiny random fluctuation of 0.00001 in val loss would reset the counter. `min_delta=0.001` requires a meaningful improvement before resetting, making training more stable.

**Why threshold 0.35 instead of 0.50?**  
In credit risk, a missed defaulter (false negative) is far more costly than a false alarm (false positive). Lowering the threshold makes the model more aggressive at flagging potential defaulters, increasing recall.

**Why save the scaler separately?**  
The API must scale incoming data using the exact same mean and standard deviation as the training data. If it recomputed scaling on each request, the model would receive differently-scaled inputs and produce incorrect predictions.

**Why stratify the split?**  
With only 22% defaults, a random split could create a test set with very few defaulters by chance. Stratification guarantees all three sets have the same class ratio, making evaluation metrics reliable.

**Why permutation importance over coefficient analysis?**  
The MLP has no interpretable coefficients. Permutation importance is model-agnostic — it measures how much performance drops when each feature is shuffled, directly answering "how much does the model rely on this feature?"

**Why two experiment phases?**  
Phase 1 (fixed experiments) tests principled combinations of init method and optimizer to understand their individual effect. Phase 2 (random search) then explores the continuous hyperparameter space more broadly. Together they give both interpretable comparisons and broad coverage.
